PaaS呓语

10-23-2018

  • 虽然不同 PaaS 平台的核心功能类似，但是针对上层的 workflow 来说，每个开发者对 PaaS 系统的理解都不相同。你的 PaaS 要给别人用，你能否理清你这个 PaaS 平台的整个工作流的逻辑？是否足够清晰？是否尽量简洁，入门容易，而不是抽象一大堆概念，堆砌一大堆所谓的功能？

  • Service服务级托管【Solaria、Opera、Kubernetes】

    • 线上的部署文件的全量更新
    • 一个Container，一个Service实例，多个应用可以以独立进程的形式部署在同一个Container中
    • 服务之间上线的并发性
      • 服务上线的是全量的部署包，所以服务之间不会依赖同一个模块的上线，本身可并发进行
    • 服务上线和扩容之间的并发性

  • 类Web应用模块级托管【ORP、ORCP】

    • 线上的部署文件的部分（假设部署文件由多个部分组成，每个部分称为一个模块）更新

      • 这种情况主要针对模块较多，并且模块之间有依赖关系，且使用一个共同的运行层（比如Web容器、一些基础so库等），部分模块更新频繁（比如业务逻辑相关的模块），且不同模块可由多个RD维护上线，运行层更新很少，如果每次模块上线都使用一个全量的包，则上线流程可能比较慢（某个模块A的上线需要依赖其他很多模块的某个版本，所以为了保证上线包的一致性，打包必须是串行的）

      • 这种模式本质的复杂性在于，具有依赖关系的且分多人维护的多模块的上线，如何在保证线上部署文件的最终一致性（某个模块上线后在线上只有一个版本）的同时，为业务上线提供最大的并发性

      • Docker Image能否解决这个问题？

        • 答案是不能，Docker Image虽然提供了分层的方式，但是本质上也是全量包，一次上线也是全量包上线，所以对于某个模块的上线，业务RD依然需要准备好所有依赖的模（即使用程序自动化，也需要指定某个模块的版本）块，使用代价太大

    • 一个Container，一个Web容器，多个服务部署到同一个Web容器中，一个服务可能对应到多个模块（一个模块通常对应于一个代码库、Web容器的一个部署目录）

    • 服务（以及模块）之间上线的并发性

      • 同一个Web容器的多个服务以及同一个服务多个模块之间可以并发上线，并且每个模块有自己独立的上线版本，同一个模块同一时间只能上线一个版本，不能多个版本同时上线
      • 通常情况下，一个服务可以对应到一个模块，这样业务的上线通常是独立的，但是有时多个业务（比如业务A、B）可能依赖一个公共模块（比如C），此时业务A和B都需要上线模块C，则这种情况下需要互斥，保证模块C只有同一个版本在上线，所以此中情况下A和B的上线无法并发进行

    • 应用（以及模块）上线和扩容之间的并发性

      • 为什么有这个需求？
        • 对于规模较小的公司来说，扩容和上线可能都是RD自己操作，可以自己协调，扩容和上线互斥是保证线上版本最终一致的较好的方法
        • 对于规模较大的公司，以及多模块的场景来说，由于上线是很频繁的，并且可能由不同业务的RD发起，而扩容可能由不同业务的OP发起，扩容阻塞上线或者上线阻塞扩容都是不太友好的，效率较低。这种情况下，如何保证上线和扩容并发进行的同时，线上的版本达到最终一致？
          • 显然扩容不会做文件的变更，只有上线才会，并且扩容拿到的部署版本一定是之前上线过的，即扩容版本Vc <= 上线的最新版本Vo，所以扩容需要不断检查上线历史，并和上线的最新版本保持一致，所以要么是扩容检测上线历史，要么是上线通知扩容
            • 扩容检测上线历史，所有扩容开始以后处于上线成功的都需要按照顺序依次追加部署，处于上线进行中的需要等待检查上线的成功与失败状态，如果是成功的则追加部署，否则忽略。【OXP的追单部署方式】
              • 全量包
              • 增量包
            • 上线通知扩容，类似分布式事务的方式，如果没收到扩容的确认消息，则不断拿着上线的版本通知扩容确认
              • 全量包
              • 增量包
